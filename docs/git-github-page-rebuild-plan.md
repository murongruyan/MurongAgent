# Git / GitHub 页面重构规划

## 1. 目标

本次重构不再以“先选本地项目目录才能看到 Git 状态”为前提，而是把 Git 与 GitHub 页面拆成两种主模式：

1. 未选择本地项目时，页面仍然可用，并直接围绕当前 GitHub 账号下的仓库展开。
2. 已选择本地项目时，页面优先展示本地仓库状态，再提供统一入口进入 GitHub 项目视图。

最终目标：

- 不再出现“已登录 GitHub 账号但因为没选本地目录而无法识别状态”的割裂体验。
- GitHub 远端仓库浏览、工作流、Release、README 查看成为一级能力。
- 本地 Git 改动、暂存、同步、推送拉取记录成为已选择本地项目后的增强能力。
- 对“可直接编辑源码”和“跨仓库修改需确认”建立清晰的权限与交互规则。

## 2. 总体导航方案

### 2.1 顶部入口

页面顶部先放一组“左上角返回样式”的功能按钮，采用两个一排的卡片式按钮布局。

第一版建议按钮：

- 返回
- 项目
- 工作流
- Release
- README
- 任务

说明：

- 未选择本地项目时，这些按钮默认作用于“当前浏览中的 GitHub 仓库”。
- 已选择本地项目时，“项目”按钮进入 GitHub 项目视图；Git 本地状态仍保留在当前页上方。
- 按钮采用统一的返回样式语言，但功能上不是都执行返回，而是作为二级入口卡片。

### 2.2 页面层级

整体拆为三级：

1. 一级页：Git / GitHub 总入口页
2. 二级页：仓库详情页
3. 三级页：工作流列表 / 单个工作流详情 / Release 列表 / README 详情等

## 3. 未选择本地项目时的显示方案

### 3.1 一级页

未选择本地项目时，页面直接显示当前账号下的所有仓库，一行一个。

每个仓库项至少显示：

- 仓库名
- 是否可直接编辑
- 是否已被选为当前新任务仓库

交互规则：

- 点击仓库名：进入仓库二级详情页。
- 长按仓库：弹出操作菜单。

长按菜单第一版包含：

- 设为新任务仓库
- 查看仓库
- 标记为可直接编辑

编辑权限规则：

- 当前被选中的仓库，模型可以直接修改源码。
- 查看其他仓库不受限制。
- 若要修改不是当前选中任务仓库的其他仓库，必须弹出用户确认。
- 仓库列表中要明确显示“可直接编辑”标识，避免误改。

### 3.2 仓库二级详情页

点击仓库后进入详情页，结构按固定顺序展示：

第一行：

- 用户名 / 仓库名

第二行：

- 仓库简介
- 长按可编辑修改简介

第三行：

- 仓库权限信息与统计
- 是否私有 / 公开
- 星标数
- 复刻数

第四行：

- 可补充默认分支、最近更新时间、语言等辅助信息

第五行：

- 操作区
- 工作流
- Release
- README

## 4. 工作流方案

### 4.1 工作流列表页

点击“工作流”后，显示该仓库所有工作流。

每个工作流项显示：

- 工作流名
- 状态
- 最近运行结果
- 最近运行时间

交互：

- 点击工作流：进入该工作流的运行列表或直接进入最近运行详情。
- 若该工作流支持手动运行，页面右下角显示悬浮加号按钮。

### 4.2 运行工作流弹窗

点击右下角加号后，弹出“配置工作流”弹窗。

弹窗内容：

第一行：

- 可切换分支

第二部分：

- 工作流自定义参数表单
- 根据 workflow_dispatch 的 inputs 动态生成

底部按钮：

- 确认运行
- 取消

### 4.3 工作流运行详情页

工作流运行后，进入单次运行详情页，按固定顺序展示：

第一行：

- 用户名 / 仓库名 / 工作流名

第二行：

- 序号
- 即第几个工作流运行，或 run number

第三行：

- 触发方式
- 例如 push / workflow_dispatch / schedule / pull_request

第四行：

- 当前状态
- 持续时间

第五行：

- 作业列表

作业展示要求：

- 显示 Job 名，例如 `build`
- 显示在多少分多少秒成功或失败
- 若正在运行，则显示进行中的耗时

### 4.4 日志交互

点击某个 Job，例如 `build`，进入日志视图。

运行中的行为：

- 自动展开当前正在运行的日志
- 当流转到下一个步骤时，自动折叠上一个步骤

已完成时的行为：

- 默认全部折叠
- 用户手动展开任意步骤查看日志

### 4.5 产物与重跑

若工作流运行成功：

- 在当前运行详情页显示产物列表
- 点击产物可下载

页面右下角固定显示“重新运行”按钮：

- 点击后复用“配置工作流”弹窗
- 允许修改分支和参数后重新运行

## 5. Release 方案

在仓库详情页中，操作区下方显示 Release 入口。

点击后进入 Release 列表页：

- 显示所有 Release
- 展开后显示产物列表

每个 Release/产物支持：

- 查看
- 编辑
- 下载
- 删除

目标效果：

- 交互和信息结构尽量接近 GitHub 网页版
- 保持移动端展示紧凑但不删关键信息

## 6. README 方案

在仓库二级详情页中，Release 下方直接显示 `README.md` 内容。

要求：

- 默认渲染 Markdown
- 支持图片、链接、代码块
- 长文支持折叠 / 展开或按段落懒加载

## 7. 已选择本地项目时的显示方案

## 7.1 单仓库方案

如果当前只识别到一个本地仓库，则页面结构如下：

第一行：

- 显示“更改:”

第二部分：

- 列出所有已更改文件名
- 每个文件名右边显示“暂存”按钮

列表底部操作：

- 全部暂存
- 全部同步

再下面：

- 推送拉取记录

再下面：

- 显示“项目”按钮
- 点击后进入前面“未选择本地项目”的 GitHub 仓库详情方案

说明：

- 本地 Git 区块负责管理工作区改动
- GitHub 区块负责远端浏览、工作流、Release、README

## 7.2 多仓库方案

如果当前项目目录下识别到多个仓库，则按仓库分组展示。

每个仓库块结构：

第一行：

- 项目名 / 仓库名

第二行：

- 更改

后续内容与单仓库方案一致：

- 更改文件列表
- 单文件暂存
- 全部暂存
- 全部同步
- 推送拉取记录

然后继续显示下一个仓库，直到全部展示完。

## 8. 没有 `.git` 但疑似仓库的方案

对于没有 `.git` 但疑似为仓库的目录，页面单独展示为“疑似仓库”。

每个疑似仓库项显示：

- 文件夹名
- 目录层级信息
- 是否包含 `README`、`build.gradle`、`package.json`、`.github/workflows` 等特征

交互规则：

- 长按：可改为子文件夹继续识别
- 右侧按钮：初始化

第一版初始化建议流程：

1. 确认是否在当前目录初始化 `.git`
2. 可选是否自动创建初始提交
3. 可选是否绑定远端仓库
4. 初始化后重新进入单仓库或多仓库模式

后续可完善能力：

- 智能推荐最可能的子仓库路径
- 自动识别 monorepo 中的独立子项目
- 初始化时引导设置默认分支和远端地址

## 9. 权限与任务规则

### 9.1 新任务仓库

长按仓库后可以设为“新任务仓库”。

设为新任务仓库后：

- 模型默认可直接编辑该仓库源码
- 页面上显示“可直接编辑”标记
- 对当前任务上下文优先提供该仓库的文件、工作流、Release 与 README

### 9.2 跨仓库修改确认

对未被设为当前任务仓库的其他仓库：

- 查看不受限制
- 修改前必须手动确认

确认弹窗建议显示：

- 当前准备修改的仓库名
- 与当前任务仓库是否一致
- 本次修改的目标文件范围

## 10. 页面实现顺序

建议按以下顺序分阶段实现：

### 第一阶段：先把页面骨架跑通

- 重做顶部按钮区，两列布局
- 区分“未选择本地项目”和“已选择本地项目”两种主模式
- 建立仓库列表页与仓库详情页基础骨架

### 第二阶段：补 GitHub 远端能力

- 当前账号全部仓库列表
- 仓库详情基础信息
- README 展示
- Release 列表与产物操作

### 第三阶段：补工作流全链路

- 工作流列表
- 运行弹窗
- 运行详情
- Job 展示
- 日志展开/折叠策略
- 产物下载
- 重新运行

### 第四阶段：补本地 Git 能力

- 单仓库改动列表
- 多仓库改动聚合
- 单文件暂存
- 全部暂存
- 全部同步
- 推送拉取记录

### 第五阶段：补疑似仓库与权限控制

- 疑似仓库识别
- 子文件夹切换
- 仓库初始化
- 新任务仓库
- 跨仓库修改确认

## 11. 当前结论

本规划确定两个核心原则：

1. 没有选择本地项目时，GitHub 页面也必须完整可用。
2. 选择了本地项目后，本地 Git 状态优先显示，GitHub 项目能力通过统一“项目”入口进入。

按这个方案完成后，页面会从“必须先选目录才能用”改成“账号仓库随时可看，本地仓库选中后再增强”，这才符合实际使用预期。
